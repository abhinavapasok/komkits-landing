The device stays put

It never moves. Vertical scroll is translated into horizontal motion for a second element underneath it.

02

Specs slide past

Each card is positioned in a row; the row's translateX is set directly from scroll progress.

03

No scroll-snap needed

Because it's continuous, not stepped, it stays buttery on trackpads and reverses cleanly on scroll-up.

04

Last card settles

The rail runs out of distance right as the section's scroll budget ends, so the last spec lands centered.

Frame 0

A sequence, not a video

This is the technique behind product-rotation scrubs — dozens of still frames, stepped through by scroll position instead of played on a timer.

Mid sequence

Canvas draws the current frame

Here it's a simple generated rotation; in production it'd be a pre-rendered image sequence loaded upfront.

Final frame

Scroll speed = playback speed

Scroll fast, it flips through frames fast. Scroll slow, you can inspect every one. That coupling is the whole appeal.

What changed from set 1

The scaffold is identical — a tall section, a sticky left column, a scroll listener computing one progress value. What differs is what that value drives: a translateX on a row of cards, a per-point linear interpolation between two path strings, or an index into a frame array drawn to canvas. Once you have the progress number, almost any visual property can be bound to it.
